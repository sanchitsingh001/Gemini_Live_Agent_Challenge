# Two-Phase Pipeline: GCP + External Asset Generation

When 3D assets are generated by an **external asset generator** (~2 hours), the GCP pipeline should **not** wait. Instead: Phase 1 runs on GCP and kicks off the external generator, then exits; when assets land in your chosen storage, Phase 2 copies them to GCP and compiles the full game.

---

## Current pipeline (relevant parts)

- **run_world_pipeline.sh**: narrative_spec → world_plan → … → **3d_asset_prompts.json** → sky → **game_bundle.json** → audio. Optional **export_godot** at the end (placeholder assets only unless you staged GLBs).
- **stage_and_export_story.sh**: Expects a directory with `game_bundle.json`, `3d_asset_prompts.json`, and **assets/*.glb**. Builds entity/npc mappings, stages GLBs into `godot_world/generated/`, then runs **export_godot.sh** to produce the web build.

So the natural split is: **everything up to and including game_bundle (and audio)** = Phase 1; **“copy assets in + stage + export”** = Phase 2.

---

## Two-phase flow

### Phase 1 (GCP Cloud Run Job, ~10–15 min)

1. Run existing pipeline **up to and including** game_bundle (and audio if desired).
   - Do **not** run `export_godot.sh` here.
   - Optional: add a flag to `run_world_pipeline.sh` (e.g. `--no-export`) or call it with `RUN_EXPORT=0` so it stops after game_bundle.
2. Upload Phase 1 outputs to GCS:  
   `gs://<bucket>/output_jobs/<job_id>/`  
   (narrative_spec, world_plan, world_graph, game_bundle.json, 3d_asset_prompts.json, sky.png, audio/, etc.)
3. **Call your external asset generator** to start asset generation:
   - Input: job_id, and either the prompts (e.g. `3d_asset_prompts.json`) or a storage URL it can pull from.
   - Output: the external generator accepts the job and will write results to a known path, e.g.  
     `<some-storage>/output_jobs/<job_id>/assets/*.glb`
4. Persist job metadata (e.g. Firestore or Cloud Storage object):  
   `status: "waiting_assets"`, `aws_job_id`, `s3_assets_prefix`, `gcs_phase1_prefix`.
5. **Exit.** Cloud Run job ends; no 2-hour wait.

### When assets are ready

You need something to detect “assets for job_id are ready” and then start Phase 2. Options:

- **A) Webhook from the external generator**  
  When the external system finishes, it calls a GCP endpoint (e.g. Cloud Run or Cloud Functions) with `job_id`. That endpoint starts the Phase 2 job.

- **B) Polling from GCP**  
  A scheduled Cloud Run Job or Cloud Scheduler–triggered function periodically lists “waiting_assets” jobs and checks your storage for the presence of a “done” marker or expected GLB count for that job_id. When ready, it starts the Phase 2 job.

### Phase 2 (GCP Cloud Run Job, ~10–20 min)

1. **Download Phase 1 outputs from GCS** into the job’s workspace:  
   `gsutil -m cp -r gs://<bucket>/output_jobs/<job_id>/* <work_dir>/`
2. **Copy assets into GCP** into that workspace, from whatever storage your external generator used (for example, syncing from another bucket or service into `<work_dir>/assets`).
3. **Run the “stage + export” steps** (same as `stage_and_export_story.sh`):
   - `generate_entity_model_mappings.py` with `--game-bundle`, `--prompts`, `--out-dir`, `--assets-dir <work_dir>/assets`
   - Copy generated mappings + `game_bundle.json` + `assets/*.glb` into `godot_world/generated/`, set `.asset_stage_marker`
   - `export_godot.sh <work_dir>`
4. **Upload Phase 2 outputs** to GCS:  
   `export/web`, web zip, etc. under `gs://<bucket>/output_jobs/<job_id>/`
5. Update job metadata: `status: "completed"`, set game link (e.g. web export URL or game server URL with `output=<job_id>`).

Result: one playable game build using the EC2-generated assets, without any GCP process waiting 2 hours.

## Integration with existing scripts

| Piece | Change |
|-------|--------|
| **run_world_pipeline.sh** | Optional `--no-export` or respect `RUN_EXPORT=0` so Phase 1 can stop after game_bundle (and optional audio). Already uses `OUTPUT_DIR`; no change for paths. |
| **run_pipeline.py** (narrative_server) | Split into two entrypoints (or two jobs): `run_phase1(story, job_id)` and `run_phase2(job_id)`. Phase 1: run spec + world pipeline (no export), upload to GCS, call an external asset generator, exit. Phase 2: download GCS outputs and external assets (e.g. from another bucket), run `stage_and_export_story.sh`-equivalent, upload exports to GCS. |
| **External asset generator** | Accepts “start asset gen” request (job_id + prompts or GCS URL). Writes GLBs under a known `assets/` path for that job. When done, call GCP “Phase 2” webhook or write a “done” marker for polling. Optionally upload directly to GCS to simplify Phase 2. |
| **Job status** | Extend job record: `waiting_assets` with external job identifiers and storage prefixes. When Phase 2 completes, set `completed` and game link. |

---

## Summary

- **Yes**, a long-running external asset generator can be integrated: Phase 1 runs the existing pipeline up to game_bundle, uploads to GCS, calls the external asset generator, and exits. No waiting on Cloud Run.
- **When assets are ready**: either sync them into GCP in Phase 2 (into the job workspace) or have the external system write them directly to GCS so Phase 2 only pulls from GCS.
- **Compile the full game** in Phase 2 by reusing the existing “stage + export” flow: `generate_entity_model_mappings.py` + stage GLBs into `godot_world/generated/` + `export_godot.sh`, then upload the resulting builds to GCS and mark the job complete with a game link.
